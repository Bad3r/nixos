# Architecture

mpv is composed in two cooperating layers: a NixOS app module that manages
system-wide install and a Home Manager app module that writes the user-level
configuration. A third orthogonal layer, the `media-toolchain` bundle, can
install mpv even when the dedicated app module is disabled. This page explains
the layering, the namespaces involved, and the gating contract that connects
them.

## Layered Composition

- **NixOS app module**
  - Location: `modules/apps/mpv.nix`
  - Namespace: `flake.nixosModules.apps.mpv`
  - Role: Adds `pkgs.mpv` and a small set of extras to `environment.systemPackages` when `programs.mpv.extended.enable` is true.
- **HM app module**
  - Location: `modules/hm-apps/mpv.nix`
  - Namespace: `flake.homeManagerModules.apps.mpv`
  - Role: Writes `mpv.conf`, `input.conf`, an `xdg.configFile` Lua hook, and a script list when the NixOS layer is enabled.
- **Media bundle**
  - Location: `modules/apps/media-toolchain.nix`
  - Namespace: `flake.nixosModules.apps.media-toolchain`
  - Role: Installs `pkgs.mpv` plus codecs and helpers. Independent of the dedicated mpv enable bit.

## Aggregator Namespaces

There are two namespaces:

- `flake.nixosModules.apps.mpv`
- `flake.homeManagerModules.apps.mpv`

For background on how the two namespaces behave differently see
[../architecture/03-nixos-modules.md](../architecture/03-nixos-modules.md) and
[../architecture/04-home-manager.md](../architecture/04-home-manager.md).

## Option Namespace

The NixOS module declares `programs.mpv.extended` with the standard options
this repo uses for app modules:

- `programs.mpv.extended.enable`
  - Type: `bool`
  - Default: `false`
  - Purpose: Master switch. Controls system install and HM gating.
- `programs.mpv.extended.package`
  - Type: package option
  - Default: `pkgs.mpv`
  - Purpose: The mpv package to install.
- `programs.mpv.extended.extraScripts`
  - Type: list of packages
  - Default: `[ pkgs.mpvScripts.thumbfast ]`
  - Purpose: Adds packages to `environment.systemPackages`. Note: see [scripts-and-bindings.md](<3. scripts-and-bindings.md>) for an important caveat.
- `programs.mpv.extended.extraPackages`
  - Type: list of packages
  - Default: `[ pkgs.mpv-shim-default-shaders pkgs.open-in-mpv ]`
  - Purpose: Additional mpv-related tools added to system packages.

The HM layer does not introduce a parallel option namespace; it consumes the
NixOS `enable` flag through `osConfig` (see below) and otherwise drives the
upstream Home Manager `programs.mpv` module directly.

### Package install contract

Both layers install mpv: the NixOS module via `environment.systemPackages` and
the HM module unconditionally via `home.packages`. HM's `programs.mpv.package`
option is typed `types.package` (not nullable), so setting it to `null` to
suppress the user-profile entry is not valid. In practice this is benign: both
sides default to `pkgs.mpv`, which is the same derivation and therefore the
same store path. The result is two profile symlinks to one store entry, not two
builds.

## Relation to the Media Bundle

`modules/apps/media-toolchain.nix` is an orthogonal app module that installs
`pkgs.mpv` along with `ffmpeg-full`, `imagemagick`, `ghostscript`, and the
GStreamer plugin sets. It is enabled on hosts that want a broad codec
toolchain. Because it adds `mpv` directly to `environment.systemPackages`, the
mpv binary is present even on hosts where `programs.mpv.extended.enable` is
`false`. In that scenario the user runs upstream-default mpv with no managed
configuration files. See [integrations.md](<4. integrations.md>) for the
implications.
